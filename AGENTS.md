# Pantheon 專案規則

本檔適用於整個 Pantheon repository。除共用 AI 規則外，所有從本專案建立的主任務、子任務與 worktree 都必須遵守以下契約。

## 可見新對話筐派工契約

當使用者說「開新對話筐」、「開新的對話」、「開卡去其他對話」、「另開任務」或明確要求在側邊欄觀看工作流程時：

1. 必須使用 Codex 的正式 thread 建立能力，建立側邊欄可見的新任務。
2. 禁止用隱藏 sub-agent、臨時 agent context 或背景代理冒充可見的新任務。
3. 建立前必須先在主工作區建立實體 `.md` 派工卡，鎖定任務目的、可改檔案、禁止範圍、驗證與交付格式。
4. 新任務的 prompt 必須包含完整派工契約；不得只提供主工作區內未提交檔案的路徑，避免乾淨 worktree 無法讀取卡片。
5. `create_thread` 只回傳 `clientThreadId` 時，狀態只能回報「排隊建立中」，不得宣稱任務已成功建立或已開始工作。
6. 必須輸出 `::created-thread` 指令讓使用者能在側邊欄看到任務，並在可查詢後核對正式 thread ID、標題、狀態與 worktree 路徑。
7. 任務完成後不得自行封存、刪除或收掉可見對話，除非使用者明確要求。

## Worktree 隔離規則

1. 多個可見任務平行工作時，預設各自使用獨立乾淨 worktree，並為每張卡分配互不重疊的唯一輸出檔。
2. 主線負責整合；派工任務不得同時修改 registry、共用 metadata、生成頁、sitemap、feed、redirects 或其他共享整合檔，除非派工卡明確指定唯一擁有者。
3. 不得預設用 `startingState: working-tree` 搬運整個 dirty working tree。
4. 若任務確實依賴未提交變更，先列出最小必要文字檔與資料契約，透過完整 prompt、文字 handoff 或明確允許的最小檔案交付；不得連同整個工作區複製。
5. 與任務無關的 binary 檔、PNG、截圖、瀏覽器證據、暫存輸出與大型 artifacts 不得帶入新 worktree。
6. 若 binary 是任務必要輸入，派工卡必須明列檔名、用途與傳遞方式；不得依賴文字 patch 隱式搬運 binary。

## 建立與失敗判定

1. 新任務建立後必須驗證：側邊欄可見、正式 thread 可查詢、worktree 初始化成功、派工卡契約可讀。
2. 初始化日誌出現 binary patch 無法套用、worktree 建立失敗、派工卡缺失或正式 thread 查不到時，判定為尚未建立成功。
3. 同一建立方法因相同 blocker 失敗三次後立即停止，不做第四次；改採不同且更安全的方法前先向使用者說明。
4. 任務尚未成功建立時不得開始產文、不得回報平行任務已在執行。

## 主線驗收

1. 每張卡完成後，主線先驗證輸出數量、唯一性、內容契約、禁詞、重複句與語法，再整合共享檔案。
2. 整合後執行受影響測試、生成器、`git diff --check` 與必要的瀏覽器驗收。
3. 回報必須區分：排隊中、已建立、執行中、已交付、已整合、已驗收；不得用模糊的「完成」涵蓋不同階段。
