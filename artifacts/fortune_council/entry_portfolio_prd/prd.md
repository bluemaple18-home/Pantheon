# Pantheon Entry Portfolio PRD

任務ID：CARD-FORTUNE-ENTRY-PORTFOLIO-PRD-001
狀態：ready-for-product-review
範圍：產品需求，不開發前端/API/runner

## 1. 問題陳述

Pantheon 目前同時具備 64 人格、塔羅插槽、八字/紫微命盤能力，但三者回答的使用者問題不同。若一開始把它們硬串成同一條 onboarding 流程，產品會變重、訊號混亂，也會再次讓需求早於產品策略被做成畫面。

本 PRD 要解決的問題是：

```text
Pantheon 第一階段應該如何定義 64 人格、塔羅、命盤/命書三個入口？
三個入口是否都可以作為第一入口？
本輪若要節省成本，應優先驗證哪個入口？
每個入口服務哪個飛輪節點？
哪些內容明確不進第一版？
```

## 2. 產品目標

### BRS-001：建立可驗證的入口 portfolio

Pantheon 需要一組清楚的入口地圖，讓 PM 可以判斷不同使用者情境下哪個入口可以先出現，以及本輪若要節省成本應優先驗證哪個入口，而不是直接進工程開發。

### BRS-002：降低第一版範圍與風險

第一階段必須避免把 64 人格、塔羅、命盤全部塞進一條重流程，降低內容、信任、成本與開發風險。

### BRS-003：保留後續商業飛輪

入口設計必須能接回外部曝光、免費入口、命中感、Discord、每日儀式、深問題、付費命書/追問、分享擴散。

## 3. 入口定位

| 入口 | 使用者問題 | 產品定位 | 飛輪節點 | 第一階段狀態 |
|---|---|---|---|---|
| 64 人格 | 我是什麼類型？我和別人差在哪？ | identity / share / community role | 外部曝光、免費入口、分享、Discord 角色 | 可獨立作第一入口；本輪低成本優先驗證 |
| 塔羅 | 我現在這件事怎麼看？今天要留意什麼？ | daily ritual / current question | 外部曝光、每日儀式、Discord 互動、回訪 | 可獨立作第一入口；適合問題導向流量 |
| 命盤/命書 | 我的長期底色與人生節奏是什麼？ | depth / paid report | 外部曝光、深問題、付費啟動、命書與追問 | 可獨立作第一入口；適合高意圖使用者 |

## 4. 入口關係

三個入口都可以作為第一入口，不是固定的第一、第二、第三關係。差別在於它們抓住的使用者起心動念不同：

| 入口 | 最適合抓住的第一念頭 | 何時適合作第一入口 |
|---|---|---|
| 64 人格 | 我想知道我是什麼樣的人 | 需要分享、社群身份、低摩擦探索時 |
| 塔羅 | 我現在有一件事想問 | 使用者帶著當下問題、情緒或日常儀式感進來時 |
| 命盤/命書 | 我想看我的人生底色/命 | 使用者有高意圖、願意提供出生資料、想看深度報告時 |

因此，本 PRD 不把 64 人格定義成唯一第一入口，也不把塔羅或命盤降級成第二入口。它們是三個可獨立啟動的入口類型。

## 5. 本輪驗證優先建議

若本輪只能選一個入口做最低成本驗證，建議優先驗證 64 人格 identity / share。

理由：

- 既有 TASK-002 已完成 64 分支、low-margin 降級、UnifiedReport 回流。
- 不需要依賴出生時間，使用者進入成本最低。
- 最容易形成分享與社群身份。
- 可作為 Discord 角色、Threads 擴散、後續命盤/塔羅入口的共同語言。

但第一版不得把 64 人格包裝成正式心理測驗，也不得讓它覆蓋八字/紫微命盤結論。

這是驗證優先順序，不是產品入口階級。

## 6. 塔羅入口定位

塔羅可以獨立作為第一入口，尤其適合「我現在有一件事想問」的使用者。

理由：

- 塔羅比 64 人格更有儀式感，適合問題導向、回訪與日常互動。
- 它能補 Pantheon 的「每日互動」與 Discord 話題。
- 它不需要先解決完整命盤交付。

第一階段若選塔羅作優先驗證，只驗證「每日/單題反思是否帶來回訪與提問」，不做大型牌陣、不做命運保證、不做完整付費抽牌服務。

## 7. 命盤/命書入口定位

命盤/命書也可以獨立作為第一入口，尤其適合高意圖使用者：已經想看人生底色、流年節奏、關係/職涯深題，且願意提供出生資料。

但在本輪低成本驗證中，命盤/命書先定位為深度層與付費層，原因是輸入成本與承諾感較高。

第一階段只需保留以下產品承諾：

- 64 人格讓使用者先看見自己的偏好與身份語言。
- 塔羅讓使用者回來問當下問題。
- 命盤/命書承接更深的長期問題與付費報告。

命盤/命書不得在第一階段被寫成「完整自動交付已準備好」。

## 8. 使用者故事

### US-001：64 人格身份入口

身為對自我理解有興趣的新使用者，  
我想先用低摩擦方式得到一個可分享的人格結果，  
以便判斷 Pantheon 是否懂我，以及我是否願意繼續探索。

驗收條件：

```text
Given 使用者第一次接觸 Pantheon
When 使用者完成 64 人格入口
Then 使用者應得到一個可理解、可分享、非診斷的人格定位

Given 分數接近中線或信心不足
When 產出結果
Then 文案必須降級成「傾向待確認」，不得輸出強分類
```

### US-002：塔羅日常回訪入口

身為已對 Pantheon 有初步興趣的使用者，  
我想用塔羅問一個當下問題，  
以便得到一個可反思、可回來追蹤的日常提醒。

驗收條件：

```text
Given 使用者提出當下問題
When 使用者進入塔羅入口
Then 產品應只回應該問題的反思角度，不回推人格或命盤因果

Given 解讀結果產生
When 文案給出建議
Then 文案不得使用保證、注定、不做會出事等語氣
```

### US-003：命盤/命書深度層

身為已經產生深問題的使用者，  
我想知道自己的長期底色與人生節奏，  
以便判斷是否值得購買完整命書或追問服務。

驗收條件：

```text
Given 使用者表現出深度探索意圖
When 產品介紹命盤/命書
Then 產品應清楚說明這是深度層或付費層，不假裝免費入口已交付完整命書

Given 八字/紫微仍是 MVP scaffold
When 文案提到命盤結論
Then 強結論必須有 evidence / confidence，不足時改成反思問題
```

### US-004：PM 入口決策

身為 PM，  
我想看到三個入口的定位、成功指標與範圍外清單，  
以便決定本輪優先驗證哪個入口，而不是直接開發功能。

驗收條件：

```text
Given PM 閱讀本 PRD
When PM 要開下一張卡
Then 下一張卡應該是驗證實驗卡或內容策略卡，不是前端/API/runner 卡
```

## 9. 成功指標

| 入口 | 第一階段指標 | 判定方式 |
|---|---|---|
| 64 人格 | 完成率、分享意圖、Discord 角色意圖、結果命中感 | 使用者測後回饋、分享點擊、進群意圖問卷 |
| 塔羅 | 回訪意圖、每日使用意圖、問題提交意圖 | 使用者是否願意明天再抽、是否想把問題帶到 Discord |
| 命盤/命書 | 付費意圖、深問題數量、想看完整報告比例 | 使用者是否願意留下問題、是否願意看命書 offer |

第一階段不要用收入當唯一成功指標；先驗證入口吸引力與後續意圖。

## 10. 範圍外

- 不做前端畫面。
- 不做 API。
- 不做 council runner。
- 不做常駐 agent team。
- 不做完整命書自動交付。
- 不做大型塔羅牌陣。
- 不做人類圖產品化。
- 不決定價格表。
- 不寫銷售頁。

## 11. 風險邊界

- 64 人格不可宣稱正式心理測驗或診斷。
- 塔羅不可宣稱事件必然發生。
- 命盤不可在 MVP 算力下輸出過度肯定的強結論。
- 所有入口都不得提供醫療、法律、投資等專業建議或保證結果。
- 不得使用恐嚇式銷售或焦慮誘導續費。

## 12. 決策

```text
產品入口關係: 64 人格、塔羅、命盤/命書都可獨立作第一入口
本輪優先驗證: 64 人格 identity/share
可替代優先驗證: 塔羅 daily ritual，如果 PM 想先抓當下問題與回訪
高意圖入口: 命盤/命書，可作第一入口，但本輪先不做完整交付
下一步: 開驗證實驗卡，不開技術卡
```
