# Entry Portfolio Council Team Roster

任務ID：CARD-FORTUNE-ENTRY-PORTFOLIO-PRD-001
用途：定義本輪臨時 council 團隊。這不是常駐 agent team，也不是實體人員指派表。

## 團隊狀態

```text
團隊型態: 臨時 council
是否已組好角色: yes
是否已指派真人 owner: no
是否需要常駐 agent team: no
PM 預設看什麼: final brief / PRD / handoff card
PM 不看什麼: 完整角色長篇討論
```

## 本輪角色

| 角色 | 本輪責任 | 要交付的判斷 |
|---|---|---|
| Product Lead | 收斂入口順序與飛輪接點 | 64 人格、塔羅、命盤/命書是否並行、串接或分階段 |
| 64 Personality Lead | 評估 64 人格作為獨立第一入口 | 是否能產生命中感、分享感、Discord 角色意圖 |
| Tarot Lead | 評估塔羅作為獨立第一入口 | 是否適合作 daily ritual / current question，並抓住當下問題流量 |
| Destiny Chart Lead | 評估命盤/命書作為獨立第一入口與深度層 | 是否適合高意圖使用者，以及本輪是否因成本先延後完整交付 |
| Community / Offer Lead | 評估 Discord 與後續 offer | 哪個入口最自然接社群、追問與命書意圖 |
| Skeptic / Risk Lead | 反對與風險檢查 | 是否混成大雜燴、過度承諾、恐嚇式銷售 |
| Cost Governor | 控制討論成本 | 是否停在產品 brief，不升級工程與 runner |

## 下一步真正要做什麼

```text
要做:
讓上述 council 角色審 PRD 與 experiment_plan，產出一份 validation brief。

不要做:
不要外部訪談。
不要手動做畫面。
不要內容測試投放。
不要開前端/API/runner。
```

## Council Validation 輸出格式

```text
1. 入口順序是否成立:
2. 三個入口是否都可獨立作第一入口:
3. 64 人格作第一入口的最大疑點:
4. 塔羅作第一入口的最大疑點:
5. 命盤/命書作第一入口的最大疑點:
6. 哪個假設最需要先被驗證:
7. 是否允許開下一張產品驗證卡:
```
